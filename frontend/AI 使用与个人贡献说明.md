# AI 使用与个人贡献说明

## 一、开发整体情况

这次做前端代码开发，其实是先把后端 API 接口都弄好了之后才开始的。最开始我自己瞎琢磨着写了点零碎的东西，比如几个输入框、按钮，还有简单的请求和数据展示的逻辑，算是搭了个最基础的架子。后来觉得自己写的又乱又慢，就用 AI 生成了大部分前端代码，把我之前写的那些零碎的内容都替换掉了。第一次提交的时候，我也没考虑太多，直接把所有前端代码都塞到后端源文件的 static 目录里了，这次重新整理，不光把前后端的代码分开了，还把原来的选项式 API 改成了组合式 API，因为 AI 生成的代码里样式都弄好了，我就没再用 Element plus，还是沿用了之前的样式。

## 二、我自己做的主要工作

虽然大部分代码是 AI 生成的，但我也不是直接照搬，还是花了不少心思去调整和优化，主要做了这些事：

1. **统一封装请求（request.js）**：在拆取成组合式API时，我把所有访问后端的请求都抽到 request.js 里了。首先用 axios 建了个统一的请求实例，把接口前缀设成了 '/api'，超时时间也统一设成 10 秒，这样不用每个请求都单独写这些配置了。

   

2. **加了请求拦截器**：我在 request.js 里加了请求拦截的逻辑，就是每次发请求的时候，自动从本地存的信息里拿 token，然后带到请求头里发过去。之前都不知道前端还有拦截器，也是看 AI 代码的时候学到的，这么一改就不用每个接口手动加 token 了，方便很多。

   

3. **处理响应拦截**：响应拦截这块主要是处理各种异常情况。比如后端返回的数据，我会先看返回码是不是 200，不是的话就统一提示请求失败；要是碰到 401 的情况，就知道是登录过期了，会自动把本地存的 token、用户信息这些清掉，然后跳转到登录页。这些是在测试的时候发现的，比如考虑到用户要是登录失效了还在操作，肯定会懵，这么处理就能避免这种情况。

   

4. **抽取公共组件，提高代码复用率**

   我发现 AI 生成的代码里，有些功能在很多页面都重复出现，于是我用AI把它们抽成公共组件，不用每个页面都写一遍：

- 把**全局提示信息**（成功提示、失败提示）抽成公共提示组件；
- 把**报修单展示、详情弹窗**单独抽成一个组件，哪个页面要用直接引入；
- 这样代码更少、更干净，后面修改只需要改一处，所有页面同步生效，大大提高复用性和可维护性。

5. **调整代码结构**：我一点一点拆，把原来的选项式 API 改成组合式 API。还把访问后端的路径都集中放到 API 模块里，每个接口封装成单独的方法，这样后面看代码的时候，一眼就知道哪个方法是干嘛的，改起来也方便，不用在一堆代码里找。另外，把前后端代码分开存放，也是我按照规范重新整理的，比第一次直接塞到后端目录里规整多了。

## 三、用 AI 的收获和自己的感悟

说实话，这次大部分前端代码都是 AI 帮着生成的，省了我不少写基础代码的时间，但我也花了很多时间去消化AI写的代码：

​	最开始我根本不知道前端还有请求拦截器、响应拦截器这种东西，可以把重复的逻辑都抽出来统一处理。（这应该算是前端**面向切面编程**的体现吧）

​	AI 的代码里会考虑到各种特殊情况，比如请求超时、返回码不对、登录失效这些，我跟着改的过程中，也慢慢养成了习惯，写代码的时候不再只想着 “正常用的时候怎么样”，还会想 “要是出问题了该怎么处理”。比如之前写代码，只要能正常请求到数据就行，现在会特意加判断，要是返回不对，得给用户提示，要是登录过期了，得自动跳走，这些细节让我觉得写代码不光是实现功能，还要考虑用的人会不会方便。最开始我写的代码东一块西一块，挺乱的，通过整理 AI 的代码，把请求封装、接口归类、API 改造这些做完，才发现代码规整之后，后面改起来有多方便。比如现在要改一个接口地址，只要在 API 模块里改一处就行，不用在各个组件里找，这也是我这次最大的体会。



## 四、总结

这次做前端开发，其实大部分代码都是靠 AI 生成的，但我没有单纯照搬，而是花了不少时间去理解、去调整。从最开始只会写点零散的组件，到后来能封装请求、处理各种异常、调整代码结构，虽然自己写的原始代码基本都被改没了，但整个过程里我真的学到了很多实用的东西，也慢慢明白写代码不光是实现功能，还要考虑后续好不好维护、用户用得舒不舒服。接下来我也想把这次学到的拦截器、代码规整的思路记下来，后面写代码的时候多用上，把这些学到的东西真正变成自己的。
